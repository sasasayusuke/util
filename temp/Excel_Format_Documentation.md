## テストケースシートの標準フォーマット

### ヘッダー行（2行目）
各テストケースシートは以下の標準的な列構成を持ちます：

| 列 | 項目名 | 説明 |
|---|--------|------|
| A | # | テストケース番号 |
| B | 大項目 | テストの大分類 |
| C | 中項目 | テストの中分類 |
| D | 小項目 | テストの小分類 |
| E | パターン | テストパターンの詳細 |
| F | userロール | 実行するユーザーの権限/役割 |
| G | 前提条件 | テスト実行前の必要な条件 |
| H | 手順 | テスト実行の具体的な手順 |
| I | 期待値 | 期待される結果 |
| J | 設計備考 | 設計に関する補足情報 |
| K | 環境 | テスト実行環境 |
| L | ステータス | テストの実行状況 |
| M | 担当 | テスト実行担当者 |
| N | 実施日 | テスト実施日 |
| O | バックログNO | 関連するバックログ番号 |
| P | 備考 | その他の補足情報 |

## 命名規則

### シート名の規則
- **共通機能**: `【共通】[機能名]`
- **ユーザー別機能**: `【[ユーザー種別]】[機能名]`

### ユーザー種別
- 共通
- 顧客ユーザー
- 支援者
- 支援者責任者
- 営業担当者
- 営業責任者
- 事業者責任者
- アライアンス担当
- アライアンス
- ソルバー

## 書き方のガイドライン

### 1. テストケース番号（A列）
- 連番で管理
- 欠番がないよう注意

### 2. 項目分類（B, C, D列）
- 大項目 → 中項目 → 小項目の階層構造
- 論理的な分類を心がける

### 3. 手順（H列）
- 具体的で再現可能な手順を記載
- ステップ番号を使用して明確化

### 4. 期待値（I列）
- 明確で検証可能な結果を記載
- 画面表示、データ変更、メッセージ等を具体的に

### 5. ステータス（L列）
- 未実施、実施中、完了、NG等の状態管理

## 運用上の注意点

1. **一貫性の保持**: 全シートで同じフォーマットを使用
2. **トレーサビリティ**: バックログNOで要件との関連を明確化
3. **進捗管理**: ステータスと実施日で進捗を可視化
4. **責任明確化**: 担当者を必ず記載
